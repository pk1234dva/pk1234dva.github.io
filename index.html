3D pathfinding 1.0.0
a 3D pathfinding solution
Overview

S.V.O. 3D pathfinding is a Unity package that offers a basic 3D pathfinding solution - this allows for pathfinding with flying enemies and other non-standard objects for which a navigation mesh approach can be too limiting. The focus of the package is to provide efficient pathfinding and search space generation that is accesible using a simple to use API. The package also contains a simple manager system that handles the scheduling of the pathfinding jobs, but you can just as well write your own.

The search space graph is implemented using a sparse voxel octree, and the path is computed using the A* algorithm. It is based on the approach described by Daniel Brewer - see his talk "Getting off the NavMesh" or article "3D Flight Navigation Using Sparse Voxel Octrees" for a more in-depth explanation.

The package does not offer any local avoidance. If you wish to use local avoidance for your flying agents, you will need to use a separate package. You can take a look at my free "orca local avoidance" package on the Unity asset store.

If you have any questions or issues, don't hesitate to contact me on discord https://discord.gg/5Mf9bbRkjz

How to start

If this is your first time using the package, I recommend to first check all the examples work:

Create a new Unity URP project using Unity 2020.3 or later, install the Jobs and Burst packages, and then install this package. Open the svo_pathfinding/Samples/sample.scene and follow the notes in the README object in the scene. After this, feel free to simply fool around and try stuff out.

The API and important components

The code in this package is separated into a few namespaces:

  • SvoPublic - intended to provide public API that should be suitable for most simple use cases.
  • SvoSamples - code examples
  • SvoDebug - various editor components for debugging
  • Svo - the largest namespace, contains various internal functions and types

This documentation mostly covers the SvoPublic namespace with a few exceptions like SvoLogger and SvoAgent. The documentation of SvoPublic elements can be seen from the 'SvoPublic' tab.

The package contains three basic components that you can find in svo_pathfinding/Runtime which should be enough for most common use cases:

  • SvoGenerateJobs - used to create the search space graph that is used by the pathfinding jobs. You can generate and save the graph inside the editor and load it at runtime, or you can generate the graph at runtime.
  • SvoManagerMono - responsible for handling queries to find paths, scheduling the path finding, and various other aspects.
  • SvoAgentBase - agent component with API for making requests to find new jobs, and also utilities that will post-process the resulting path, to make the path less choppy and more efficient.

Besides the three components above, there's also SvoVisualize inside svo_pathfinding/EditorOnlyComponents that is intended to be used for debugging purposes.

Building projects

Various messages and errors are logged using the SvoLogger class. To modify the behaviour, e.g. removing function calls, see the class documentation.

By default, Burst is enabled for all possible jobs. Either disable Burst compilation using the Job menu in the Unity editor, or delete the "#define SVO_BURST" inside the files with jobs definitions - this will cause the [BurstCompile] attribute to be ignored using preprocessing. The pathfinding jobs are in SvoJobs.cs, and the svo generation related jobs are in the files inside Internal/SvoGenerateJobsDefinitions.

Components that are meant to be used in the editor only are in the Runtime/EditorComponents. To avoid using them in builds, it's recommended to use these with GameObjects that are tagged as "EditorOnly".

Serializing svo data can be done either using the SvoDataSO scriptable object, or serializing directly into a binary format. Using binary serialization is more memory efficient, but it doesn't allow for visualization of the svo. Binary serialization also requires that you store the svo inside StreamingAssets. Binary serialization loads the svo using the path to the file - the SvoManagerMono.svoDataBinary field allows for drag and dropping a file, but internally this only updates the SvoManagerMono.svoDataBinaryName field below it using an OnValidate method - it's this string that is used to check for existence of the binary file.

I recommend using scriptable object serialization, until the point where you notice performance issues in the editor.

General software architecture

In general, the public API functions should not throw any exceptions. If any exceptions are thrown, it's unlikely they could somehow be recovered from. Failures are dealt using return values or by checking the value of some global state variable.

Various native memory deallocations are handled internally if things fail, with the exception of SvoGenerateJobs.CreateSvoSynchronous.

This documentation covers members that are intended to provide an easy to use API, and these members are mostly public. There might be some other public members that are not documented, but they are not always intended to be something that's safe to use. If you don't want to look into the source code too much and wish to avoid potential issues, I'd suggest just sticking with the API that's described in this documentation, and with the SvoGenerateJobs, SvoManagerMono, SvoAgentBase components in particular.
